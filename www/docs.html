Webjar can be easily used both with and without extending the framework. Extending requires some skills in Java to create new RESTful endpoints, and this is a topic that will be covered at some point later on in the ddocumentation. For now, let's get the application up and running.

Philosophy

The underlying philosophy of the webjar framework is to give as much power as possible to the user with managing their web content and abstract the plumbing that comes with setting up and running a web server. To this effect, the user is required to be familiar with Javascript to fetch and display content from a RESTful API. There are numerous Javascript frameworks that do this very well, and I mean really well. This means that ANY design can be plugged into this framework, and all that is required of the user is to apply the necessary bindings between the UI element and the underlaying JSON data.

Installation

Download one of the compressed artifacts (webjar-[version].bin.zip/.tar.gz]) and unpack. Find the "/webjar-[version]/dist" directory inside the unpacked folder and copy the content inside to where you will server your application from (the application root). This will contain the runnable jar together with other necessary configuration files.

For the assets, download one of the compressed artifacts assets (webjar-[version]-assets.zip/.tar.gz]) and unpack. Find the "/www" directory inside the unpacked folder and copy that folder to the application root. This will contain the content to be served by the application.

Setup

You might be planning to run this on your local machine or on a remote online server. Either way, the setup required remains the same. What you will need is:

  • Runtime Environment - the minimum jre version required is 1.7 and the application has also been tested with version 1.8
  • Application Config - the 'app-config.properties' file is at the application root. It provides sensible defaults which you can override before firing up the application for the first time. The current values are:
    • app.initialized - this should be 'false' and will be set to 'true' by the application when it first runs
    • app.name - this is currently not being used but the intention is to identify the application uniquely in the database
    • app.jdbc.database - this is a required field used for the database name when the application creates the sqlite database file
    • app.admin.username - this is a required field used by the application to create a default user who is the administrator for the site
    • app.admin.firstname - this is a required field which becomes the default admin user's first name
    • app.admin.firstname - this is a required field which becomes the default admin user's last name
    • app.admin.email - this is a required field which is the admin user's email
    • app.server.class - this is not required unless you are extending the framework in your application
    • app.port - this should be set to whatever port you want the app to listen at when using http protocol
    • app.ssl.enable - this should be set to false for the setup, and can be set to true later on when you configure you ssl certificates.
    • app.ssl.port - this should be set to whatever port you want the app to listen at when using https protocol
  • Mail Config - the 'mail-config.properties' file is at the root of the unpacked files. It provides values needed to configure smtp properties for emailing. Some properties might need to be obfusicated like the smtp username and password. To do this we need to use a special tool.
    • In the unpacked files, navigate to 'tools\bin'
    • Execute the following to encrypt any property you'd like to:

      $ ./encrypt.sh input="This is my message to be encrypted" password=MYPAS_WORD verbose=false

      The output will be any encrypted string like:
      uv9+BnQFuZbfTV5Kf45oBOr0eJzBW5AS+XaYY+Lu5XWYhGgl0Ee41P0QUGpIrfyD

    • As a general rule, jasypt expects encrypted configuration parameters to appear surrounded by "ENC(...)". So in the properties file, add the encrypted values in this manner:
      • mail.smtp.server=smtp.gmail.com
      • mail.smtp.username=ENC(nG/vEEcd+iccp8eCONkv1qKKc7GpK31eSF1lO/9k564=)
      • mail.smtp.password=ENC(B/56MAhS46KLoI71bbYmetCt/5cHURho)
  • Startup Options - You can see these in the "startup-options.txt" file in the application root. when starting the application, there are currently 3 values you can override using VM arguments.
    • -Dhost="" the default is the value in "app.host"
    • -Dport="" the default is the value in "app.port"
    • -DhttpsPort="" the default is the value in "app.ssl.port"
    • -DhttpsEnable="" the default is the value in "app.ssl.enable"
    • -DkeystorePath="" the default is the value in "app.ssl.keystore"
    • -DkeystorePassword="" the default is the value in "app.ssl.storepass"
    • -DkeymanagerPassword="" the default is the value in "app.ssl.managerpass"
    • -DlogsHome="" the default is the value in the "app.logs.home". This is the folder where log files will be written to
    You can also override startup options using program arguments, and the arguments expected are
    • [0] - host
    • [1] - port
    • [2] - httpsPor
    • [3] - httpsEnable
    • [4] - keystorePath
    • [5] - keystorePassword
    • [6] - keymanagerPassword
    • [7] - logsHome
    • [8] - idleTime
    • [9] - maxHeaderSize
    • [10] - maxThreads

  • If you need to enable SSL on your app, look at this blog

  • Starting - From the command prompt in the application root, run "java -jar webjar-[version]-fat.jar" and this should start the setup process with all default options. You will be prompted for a 'password' which will be encrypted and saved together with the admin credentials supplied in the properties file.

    If you wish to override any default options, the general startup syntax is:

    java [VM args] -jar webjar-[version]-fat.jar [Program args]

  • Running - When the application is successfully started, navigate to "http://[host]:[port]/home" and start using the application. The default should be http://localhost:8082

Administer

From the landing page, you can navigate to the blog using the "Blog" link on the navigation menu. From the blog home, you can navigate to the admin section using the "Admin" link on the navigation menu. You will encounter a login page which will take the username and password you used during initial setup. Once you log in successfully, you will be able to do among other things:

  • Update you profile
  • Manage the "About Us" and "Contact Us" page content
  • Manage site-wide content using key-pair values
  • Manage site media content (images)
  • View the REST API and access to the testing utility for the endpoints
  • Manage dynamic content for custom pages. Here you can
    • Manage database tables. To create tables though, the preferred method is using an SQLite client of your choice
    • Manage database queries and their associated parameters
    • Manage dynamic pages and specify bindings to database tables and queries
    • Manage the content (insert/update) the content in database tables

Developer

In some cases, a user might wish to provide custom endpoints to meet their own specific needs while using the framework. To accomplish this, it's best to just extend the framework by embedding it in your application.

If you wish to embed this framework in your application, the easiest and most convenient way is use the webjar-quickstart-archetype assessible through:

mvn archetype:generate -DarchetypeGroupId=com.jarredweb.repo -DarchetypeArtifactId=webjar-quickstart-archetype -DarchetypeVersion=0.2-beta

Otherwise, the steps below would suffice also. But remember that you also need to download and extract the application's assets from one of the compressed bundles provided (.zip/.tar.gz)

  • Include the webjar-[version].jar in your classpath. To do this, you can either:
    • Download the latest jar file (the current one is webjar-0.2-beta.jar) and place it in your classpath
    • Alternatively, if you are using maven, you can use the following maven coordinates in your pom.xml 
      
<dependency>
  <groupId>com.jarredweb.repo</groupId>
  <artifactId>webjar</artifactId>
  <version>0.2-beta</version>
</dependency>
  • The basic interface for booting up the web application is as shown below: 
    
public interface WebjarBootup {
	
	ApplicationContext initApplicationContext();

	Properties getStartupOptions(String[] args, ApplicationContext ctx);
	
	WebjarBootup initStartupService(ApplicationContext ctx);
	
	void startServer(Properties startOptions, ApplicationContext ctx);
	
	void onShutdown();
}
  • The application will first create the application context using Spring. It scans @Configuration files and uses these to build the context. While doing this, it will load the user-defined app-config.properties file which overrides default values that are shipped with app-default.properties. It also loads the mail-config.properties file. This is one step which your application should extend to load its own child context to be merged with the parent context. 
    
@Configuration
@ComponentScan({ "works.hop.webjar.ctx"})
public class AppCtxConfig {
	
	private static final Logger LOGGER = LoggerFactory.getLogger(AppCtxConfig.class);

	private static AnnotationConfigApplicationContext ctx;

	public static AnnotationConfigApplicationContext getAnnotationConfigApplicationContext() {
		if (ctx == null) {
			AnnotationConfigApplicationContext parentContext = WebjarSpringContext.getAnnotationConfigApplicationContext();
			
			ctx = new AnnotationConfigApplicationContext(AppCtxConfig.class);
			ctx.setParent(parentContext);
			ctx.setClassLoader(parentContext.getClassLoader());
			ctx.registerShutdownHook();
			LOGGER.info("{} context created", AppCtxConfig.class.getName());
		}
		return ctx;
	}
}
  • Next the application will load startup options which it first tries to read from VM arguments, or else it falls back to the program arguments (String[] args) or finally it falls back from the values loaded from the application files (in the application context).

    Next the application will start the StarupService which does the necessary house-keeping chores before starting the web server

  • Next the application will call the startServer() method which deploys the web services. To add new endpoints, you will need to add them to Jersey's configuration object. This steps will therefore require a custom task to make it happen. This is also where your custom application needs to be wired to use Spring

    Extend the AppRestConfig class to return you new custom Resource Configuration

    
public class AppRestConfig extends WebjarRestConfig {

	public AppRestConfig(final ApplicationContext ctx){
		super();
		packages("works.hop.webjar.ext");
		property("jersey.config.server.mvc.templateBasePath.freemarker", "templates/freemarker/");
		property("jersey.config.server.mvc.caching.freemarker", false);
		register(FreemarkerMvcFeature.class);
		register(new AbstractBinder() {
			
			@Override
			protected void configure() {
				AppServices services = ctx.getBean(AppServices.class);
				bind( services ).to( AppServices.class );
			}
		});
	}
}

    Your custom resources should be in the 'my-custom-package-name' package, in this example

  • Extend the AppExtServer class to return the new resource config 
    
public class AppExtServer extends AbstractAppServer{

	public AppExtServer(ApplicationContext ctx, Properties startOptions) throws Exception {
		super(ctx, startOptions);
	}

	@Override
	public ResourceConfig getResourceConfig() {
		return new AppRestConfig(ctx);
	}
}

  • The AppRunner class should be simple to follow

    
public class AppRunner extends WebjarRunner {

	private static final Logger LOGGER = LoggerFactory.getLogger(AppRunner.class);

	public static void main(String[] args) {
		AppRunner runner = new AppRunner();
		AnnotationConfigApplicationContext ctx = (AnnotationConfigApplicationContext) runner.initApplicationContext();
		Properties startOptions = runner.getStartupOptions(args, ctx);
		runner.initStartupService(ctx).startServer(startOptions, ctx);
		LOGGER.info("AppServer started");
	}

	@Override
	public ApplicationContext initApplicationContext() {
		AnnotationConfigApplicationContext ctx = AppCtxConfig.getAnnotationConfigApplicationContext();
		LOGGER.info("app context in webjar-starter created");
		return ctx;
	}
}

    • Concepts

    The basic data models used by the application revolves around a few couchbase documents and a few sqlite tables

    The basic tables are:
    • tbl_user_account
    • tbl_category
    • tbl_media
    • tbl_subscriber

    You are not just limited to these. You can be able to add more tables and bind them to pages dynamically

    The documents are:
    • SiteContent 
      
{
	"site": {
		"contactUs": {},
		"aboutUs": {},
		"dateCreated": "",
		"mapValues": {},
		"_id": "site_content",
		"type": "site_content"
	}
}
    • DynamicContent 
      
{
	"dyna": {
		"tables": {},
		"dateCreated": 1504190351184,
		"pages": {},
		"_id": "dyna_content",
		"type": "dyna_content",
		"queries": {}
	}	
}

    Site Content

    MapValues are key/value pairs which are available to every page that is returned by the application. Webjar binds these MapValues to the page on the server side using JTwig, it's default Templating Engine. The one important you should note is that to exptrapolate values from the JTwig Model object, use the ${key} syntax, instead of the default {{key}}. This makes it possible to use {{key}} syntax to work with Vuejs (and other view frameworks) in the frontend
    AboutUs and ContactUs, as you can probably easily guess, contain key/values pairs which are accessible only in the "About Us" and "Contact Us" pages respectively.

    Dynamic Content

    The Tables object contains metadata about the tables in the relation database. This is used to control attributes about form elements that are generated dynamically. When you add a new table, you should sync it up with the Dynamic content to be integrated with the application. This is done easily with the click of a button in the administration panel
    The Queries object contains user-defined SQL queries with (optionally) variable parameters. These queries can be bound to a page so that they may return data when queried from that page. This binding done easily in the administration panel
    The Pages object contains user-defined pages when a user can bind static key/value pairs, tables or queries so that these are available to the requests that come from that page. This is what gives the application its versatility and adaptibility to many use-cases.

    Principles

    The architecture of this application is inspired by a few principles with the objective of making it adaptible to a wide variety of use-cases while still able to serve its core purpose, which is serving web content. These principles are:

    • Strive to remain simple without being simplistic
    • Strive to achieve ease of use for every kind of user, from the designer to the developer
    • Strive to use the Web's basic technology alone (HTML5, CSS, JavaScript) for the View concerns
    • Strive to keep the View concerns isolated from the Application's so that they may both evolve independently
      • Easier to upgrade or update the Application
      • Design is isolated from development
      • Easier to adopt and use different designs
      • Easier to prototype and test designs
    • Strive to use a minimal set of operations in the REST API to keep it simple
    • Strive to use creative solutions without reinventing the wheel
    • Use a software platform for the Application that will ensure portability and rapid development
